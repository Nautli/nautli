## 한줄 총평
최빈 과업(리뷰 카드 답변)이 우측 상단 24px 링에 유배됐다 — 이 앱셸은 "검색 위주 노트앱" 문법으로 짜였지 "매일 큐를 소화하는 앱" 문법이 아니다.

## JTBD & 사용맥락 점검

- **진짜 job:** 아침에 열어서 오늘의 리뷰 큐를 태운다(플로우 A가 최빈이라고 스펙에 못박음). 근데 앱바 레이아웃은 **중앙 480px를 검색에 준다** — 검색은 플로우 B(간헐적). Superhuman·Things·Linear Inbox처럼 큐가 주 job인 앱은 큐 진입점이 시각적으로 지배해야 한다.
- **사용 맥락:** 개발자가 IDE ↔ nautli를 슥슥 왕복. 창 크기가 작을 확률이 높다(사이드 도킹, 스플릿뷰). 근데 스펙에 **반응형/좁은창 규칙이 0줄**. 800px 너비에서 라벨(120px)+중앙(480px)+링+숫자가 어떻게 붕괴하는지 미정.
- **키보드 우선 유저 미스매치:** 최빈 job에 전용 단축키가 없다. `⌘J`가 리뷰점프인데 `⌘1~5`에 묻혀 `?` 시트에서만 발견됨. Superhuman은 `J/K`가 큐 이동, Linear는 `I`가 인박스. 리뷰 앱이 primary 단축키를 감춰두면 안 된다.

## 결함 목록

### P0 (치명 — 지금 그대로 못 나감)

**P0-1. 리뷰 큐 진입점 시각 우선순위가 뒤집혔다**
- 왜: 최빈 job이 `24px 링 + 작은 숫자`로 우측 끝에 있다. 반면 검증적 job(검색)은 중앙 480px 지배. Fitts's Law 관점에서 24px 원형 타겟은 마우스 정확도 하한(48px 권장). 시선 우선순위(중앙>우측 코너)와도 반대.
- 수정: **중앙을 리뷰 큐 CTA로**. `오늘 12 / 28  ▶` 세로 파일 스타일 pill(높이 32px, 폭 auto ~180px), 좌측에 얇은 링, 우측에 화살표. 검색은 오른쪽으로 밀거나(작은 pill `⌘K`) **⌘K 트리거만 남기고 pill 자체 제거**(Raycast는 앱바에 검색 없음, 팔레트만). 참고: Things 3의 "Today" 뱃지, Superhuman의 Inbox 카운터.

**P0-2. violet 펄스와 진행 링이 같은 슬롯에서 mode-swap**
- 왜: "링 42% 채움" ↔ "violet dot pulse"가 동일 위치에서 교대. 유저가 아침에 링(진행)을 기대하고 슬쩍 봤는데 dot(소화중)만 있으면 "내가 어제 다 태웠나? 리셋됐나?" 인지 붕괴. 상태를 겹치는 건 UI 반패턴(SwiftUI/Material 모두 상태 채널 분리를 권장).
- 수정: **채널 분리.** 큐 진행은 항상 CTA에 유지, 소화 상태는 **왼쪽 트래픽라이트 옆 좁은 슬롯(80~160px 사이)**에 24px violet dot만 조용히. 소화 완료 토스트는 그대로.

**P0-3. 40% 잉크·대비비 접근성 실패 위험**
- 왜: 다크 배경(#16) 위 잉크 40%로 하단 힌트 바(10px), 진행 숫자 "페이드", 셀렉 안 된 메타(10px)를 깐다. WCAG AA 4.5:1은 10~13px에서 거의 확실히 미달. 개발자 유저 대부분 다크 모드 상시 = 이거 안 보인다. `이대로 가면 안 됨`이 2번째로 큰 리스크.
- 수정: 텍스트 40%는 아이콘·구분선에만. 텍스트 최소 60%(4.5:1 실측 후 확정), 10px 폰트는 앱셸에서 금지 — 최소 12px. 그리고 SSOT `DESIGN.md`에 "잉크 opacity별 사용처 매트릭스" 박아라(카드 [[디자인프레임-레퍼런스토큰화]]).

### P1 (중요 — 나가기 전 처리)

**P1-1. "현재 위치 라벨"은 삭제 대상 (열린 질문 1의 답)**
- 왜: 사이드바 하이라이트와 중복. 사이드바 접힘·전체화면은 파워유저 극단 케이스이고, 그 경우엔 오히려 `⌘.`로 사이드바 열면 그만. Obsidian·Arc·Linear 다 상단 좌측 무음. 라벨을 남기면 앱바가 항상 "글자 3개 + 그레이"로 지저분해진다.
- 수정: 라벨 삭제. 그 자리는 P0-2의 소화 상태 채널 또는 완전 무음.

**P1-2. ⌘K 팔레트는 검색+명령 통합, 단 접두어 분리 (열린 질문 2의 답)**
- 왜: 유저가 개발자 = Raycast/Linear/VSCode Command Palette가 모국어. 검색 전용은 익숙한 손 기억을 반토막. Linear 방식이 정답: 기본은 검색, `>` 접두어는 명령 모드.
- 수정: `>` 접두어 시 명령 리스트(섹션 이동·소화 지금·설정 열기·리뷰 스킵). 결과 리스트는 절대 섞지 마라 — 스펙 명시 필요. 힌트 바에 `> 명령` 힌트 노출.

**P1-3. 리뷰 0/0일 때 완전 숨김 (열린 질문 3의 답)**
- 왜: "회색 완결 링"을 남기면 시각 노이즈만 되고, 어제 밤 소화가 실패했는지 아닌지 툴팁으로만 알려주는 건 정보 은닉. 대신 **완결 시 CTA를 다른 문장으로 스왑**: `오늘 완료 · 다음 소화 03:30` (툴팁 아니고 텍스트로). 소화 데몬 마지막 실행 시각은 hover 아닌 그 자리에 always-visible로. Recognition > recall.

**P1-4. 반응형 규칙 부재 → 창 좁을 때 붕괴**
- 왜: 800px 이하에서 라벨+중앙 pill+링이 겹친다. 개발자는 앱을 사이드 도킹으로 자주 씀.
- 수정: 단계별 축소 규칙 명시 — `<960px`: 라벨 숨김. `<720px`: 중앙 pill이 아이콘만(⌘K 뱃지). `<560px`: 리뷰 CTA에서 링만 남기고 텍스트 숨김, hover/focus 시 팝오버. 각 브레이크포인트 스크린샷 3장 첨부.

**P1-5. 큐 진행이 답변수/스킵/스누즈를 구분 못 함**
- 왜: `오늘 12 / 28`은 "12 답변" vs "12 스킵/스누즈 포함"이 애매. 유저가 정말 신경 쓰는 건 실제 처리된 카드. Superhuman은 아예 "N left"만 표시(남은 것 프레이밍).
- 수정: **남은 프레이밍으로 스위치** — `오늘 16장 남음`. 마우스 hover 시 `답변 10 · 스누즈 2 · 스킵 0` 브레이크다운. 완결 상태는 `오늘 완료 ✓`.

**P1-6. 팔레트 상태 누락 — loading/0 result/API 실패**
- 왜: 스펙에 정의된 상태 = 빈(최근검색) + 결과 있음. 검색 debounce 중, 결과 0건, `/api/memory` 실패, 로컬 인덱스 재빌드 중 상태 없음.
- 수정: 4개 상태 각각 목업 필수. 0건은 `검색 결과 없음 · > 로 명령 시도` 힌트, API 실패는 `연결 실패 · 재시도` (inline retry), 로딩은 skeleton 3줄(스피너 금지 — 조용한 크롬).

**P1-7. 팔레트 선택 인디케이터 "잉크 100% 좌측 2px"가 배경 5%와 함께 오는데**
- 왜: 다크 배경 위 잉크 100% 2px 바는 밝기가 튄다(SSOT 무채색 원칙과 마찰). 반대로 배경 5%만으론 키보드 유저가 선택행을 놓친다.
- 수정: 좌측 2px 대신 **행 전체 배경 8% + 좌측 3% 헤어라인 강조**(구분선처럼). 색이 아니라 명도 대비만으로. 셀렉 행 텍스트는 잉크 100%로 승격.

### P2 (개선)

**P2-1. `⌘.` 사이드바 토글은 옵시디언과 충돌 안 하지만, macOS 관례는 `⌥⌘S`(Notes)·`⌘0`(Finder). 유저가 옵시디언 병용자면 `⌘.`가 정답. 스펙에 이 결정 이유 한 줄 남겨라 — 나중에 왜 이 키인지 재검토 방지.

**P2-2. 팔레트 결과 각 행 폰트 사이즈가 미정** — 스니펫 1줄만 있는데 잉크 90%면 몇 px? 13/14px 명시 필요.

**P2-3. 첫 실행 유저의 팔레트 빈 상태 = "최근 검색어 5개" — 신규 유저는 0개**. 대체: "검색해 볼 것들 · scope 목록 · '기억 만들기' 딥링크". 온보딩 hook 자리.

**P2-4. `prefers-reduced-motion` 대응 부재** — violet dot pulse는 vestibular 이슈. 감소 모드에선 정적 dot로.

**P2-5. 리뷰 큐 CTA 클릭이 "질문 섹션 이동 + 첫 카드 포커스"인데, 이미 질문 섹션 안에서 3번째 카드를 보고 있을 때 CTA를 누르면 어떻게 되나?** 유지 vs 첫 카드 리셋 정의 없음. 유지가 답(사용자 컨텍스트 존중).

## 업계 레퍼런스 패턴

1. **Superhuman (인박스 소화 첫째 앱셸)** — 상단 중앙에 `Inbox · 42` 크게, 검색은 `⌘K` 그림자 팔레트만. 큐가 chrome의 hero. → nautli의 리뷰 CTA도 같은 무게로 (P0-1).
2. **Linear (개발자 키보드 우선 팔레트)** — `⌘K`는 검색+명령 통합, `>` 접두어로 명령 모드. 결과 리스트에 아이콘 + 스니펫 + kbd shortcut 우측 정렬. 인디케이터는 배경 명도만(외곽 컬러 X). → P1-2, P1-7.
3. **Arc (조용한 크롬)** — 상단 브레드크럼·라벨 0, 트래픽라이트 옆은 완전 무음, 우측에 아주 작은 상태 dot만. → P1-1 "라벨 삭제" 근거.
4. **Things 3 (오늘 프레임)** — "Today"를 남은 개수 프레임(`3`)으로. 완료 시 "All caught up". → P1-3, P1-5.

## 이대로 가면 안 되는 1가지

**리뷰 큐 진입점이 검색 pill보다 작고 시각 우선순위가 낮다.**
매일 열어서 태우는 게 job인 앱인데, chrome이 "가끔 검색하는 노트앱" 문법으로 짜였다. 3주 안에 유저의 큐 소화율이 하락하고 "카드가 쌓여 있는지 몰랐다"는 피드백이 온다. 시각 우선순위를 뒤집고(P0-1) 상태 채널을 분리(P0-2)한 다음에야 팔레트 얘기가 유효하다.
