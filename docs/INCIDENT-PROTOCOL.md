# 오염·사고 대응 규약 (Incident Protocol) v1

> TASK-101 · 2026-07-23. 대상 사건: ①기억 오염(잘못된/조작된 fact 적재) ②오병합(서로 다른 사실의 병합) ③데이터 유출(내용이 외부로 전송) ④잘못된 기억의 행동 전파(오염 fact가 recall로 세션에 전달되어 에이전트 행동에 영향).
> 원칙: 1인팀 체급 — 각 단계는 체크리스트+문구 템플릿이다. 조직 규정 흉내 금지. **사고를 숨기는 것이 사고 자체보다 신뢰 해자를 더 태운다.**

## 발동 기준

다음 중 하나가 관측되면 발동한다(의심 단계에서 발동 — 확정 기다리지 않음):
- 오병합 실사례 1건 (→ [TRUST-CLAIMS](TRUST-CLAIMS.md) A3 철회 조건과 동시 발동)
- claim 본문이 telemetry/외부 endpoint로 나간 흔적
- 특정 출처(볼트 이식분·자동캡처분)의 체계적 오염 의심
- 유저(파일럿 포함)의 "기억이 이상하다" 신고 중 재현되는 것

## 1단계 — 즉시 격리 (발견 후 바로)

- [ ] 오염 의심 fact 식별: fact_id 목록 또는 출처 단위(`provenance.source`)·시간창 단위로 범위 확정
- [ ] recall 차단: TASK-096(quarantine) 구현 전까지의 **임시 조치** = 해당 fact `invalidate`(non-lossy, undo 가능) 또는 심각 시 daemon 정지(`launchctl bootout`) + MCP recall 사용 중단 안내
- [ ] 범위 스냅샷: `~/.nautli/events/` 당월 파일과 `index.sqlite`를 별도 사본으로 보존(증거 보전 — 조사 중 덮어쓰기 방지)
- [ ] 확산 확인: 감사 원장 Q2(전달 추적 — [AUDIT-LEDGER-SPEC](AUDIT-LEDGER-SPEC.md))로 오염 fact가 이미 전달된 세션 목록 확보

## 2단계 — 영향 유저 통지

통지 기준: **오염 fact가 유저 세션에 1회라도 전달됐으면 통지한다**(전달 0건이면 3단계로 직행, 공개 기록에는 남김).

문구 틀(ko — en은 동일 구조):

> [nautli] 기억 데이터 문제 안내
> {날짜}에 적재된 기억 중 {범위 요약, 예: "7/20 볼트 이식분 중 N건"}에서 {문제 유형: 오염/오병합}이 확인되었습니다.
> 영향: {이 기억이 전달된 시점·세션 수}. 해당 기억은 현재 recall에서 차단(격리)되어 더 이상 전달되지 않습니다.
> 조치 방법: {유저가 할 일 — 보통 "없음, 자동 격리됨" / 필요 시 "nautli review에서 해당 카드 확인"}
> 원인과 재발 방지는 조사 후 {링크}에 공개합니다. 불편을 드려 죄송합니다.

- [ ] 과장·축소 금지: 아는 것/모르는 것을 구분해 쓴다("원인 조사 중"은 유효한 문장)
- [ ] 통지 채널: 파일럿 단계 = 해당 유저 직접 채널(메일/디스코드). 통지 전 발송 컨펌 게이트(외부 발송 규칙) 적용

## 3단계 — 재평가

- [ ] TRUST-CLAIMS에서 이 사건이 철회 조건을 건드린 주장 전수 확인 → 해당 카피 즉시 하향/철회(사이트·README 커밋)
- [ ] 관련 수치 재측정 계획 수립(예: 오병합이면 자동병합 표본 재라벨) — 재측정 완료 전까지 해당 주장 복귀 금지
- [ ] 근본 원인 1~2개 수렴: 프롬프트/정책/코드 어느 층인지, 재발 방지가 "안 하게"가 아니라 "못 하게"(코드 가드+테스트)로 박혔는지

## 4단계 — 공개 기록

- [ ] 리포 `docs/incidents/YYYY-MM-DD-<slug>.md`에 기록: 무엇이 일어났나 / 언제 발견·격리했나 / 영향 범위(전달 세션 수) / 원인 / 조치와 재발 방지 / 철회·복귀된 주장
- [ ] 숨기지 않기: 발견 경위가 "유저 신고"였으면 그렇게 쓴다
- [ ] TRUST-CLAIMS 해당 행 갱신(철회 조건 발동 이력 병기)
- [ ] 하네스/nautli remember에 교훈 기록(내부 학습 루프)
